# 指令：架构及模块维护关系优先

目标：在自动补全、代码建议、重构与模块化讨论中，优先考虑系统架构、模块边界与维护关系（ownership）。

范围：
- 适用于整个仓库，特别是 `main/src`、`preload/src`、`renderer/src` 等模块化子包。
- 影响：代码实现、模块依赖变更、接口设计、代码注释、文档、以及 PR 描述中的决策理由。

核心规则：
1. 在提出代码修改建议前，先说明受影响的模块、模块边界、以及它们的依赖关系。示例："修改 `WindowManager` 会影响 `modules/SingleInstanceApp` 与 `modules/LoggerModule` 的初始化顺序"。
2. 优先保持或改善模块可维护性与低耦合：当有实现选项时，选择更有利于清晰边界（单一责任、最小公共接口）的方案。
3. 标注维护者/负责人（ownership）：建议在模块相关 comment 或 PR 描述中注明主要维护人或团队，若不明确则提示需要指定。
4. 在架构相关变更中，强制提供影响范围（affected modules）、迁移步骤与回滚方案草案。

行为准则（提示模板）:
- 当请求补全或重构建议时，先输出：模块列表 → 受影响的公共接口 → 推荐变更方案（架构/接口优先）→ 维护职责/迁移步骤。
- PR 描述模板示例（生成时遵循）：
  - 概要：一句话说明变更目的
  - 受影响模块：列出模块与文件路径
  - 依赖影响：列出上游/下游模块
  - 维护者：建议的负责人或需确认的团队
  - 迁移步骤与回滚：简要步骤

实施细节：
- 仅作偏好时，请以“建议/优先”表述；涉及破坏性或兼容性变更时，必须以“必须/强制”表述并要求明确迁移方案。
- 对于小型修复（纯局部变量、拼写修正等），可跳过架构说明，但仍应在 PR 中注明是否为局部修复。

示例 prompt（示范如何向 Copilot/助理提出请求）：
- "为 `modules/WindowManager` 提供一个非侵入式重构建议，优先保持现有模块边界并减少对 `OrpcModule` 的同步依赖，列出受影响模块与迁移步骤。"
- "当我请求实现新的 API 接口时，请先生成受影响模块清单与维护者建议，然后再给出代码实现。"


补充规则：选择的依赖库及模块分类优先

目标：在引入、升级或替换第三方依赖，以及对仓库模块进行分类时，优先考虑依赖的维护性、兼容性与与模块分类的一致性。

规则要点：
- 在建议新增或升级依赖前，列出备选项、维护活跃度（release 频率/维护者）、安全历史（已知 CVE/漏洞）、以及与本仓库核心模块的兼容性评估。
- 依赖选择优先级：1) 安全与维护；2) 与现有模块分类一致性；3) 轻量与性能；4) 社区与文档支持。
- 模块分类：建议在仓库中将模块归类为 `核心(Core)`、`平台(Platform)`、`集成(Integration)`、`可选(Optional)`，并在变更建议中标注每个受影响模块的类别与风险等级。
- 对影响 `核心` 或 `平台` 模块的依赖变更，必须提供回滚/降级方案与基本测试矩阵（例如针对不同 Electron/Node/Renderer 框架版本的验证要点）。
- 对于可选或实验性依赖，建议通过 feature-flag、插件化或隔离包的方式接入，并在 PR 中明确标注风险与默认行为。

行为准则（生成时）：
- 在给出依赖或模块分类相关建议前，先输出：依赖候选对比表 → 与模块分类的映射（例如该库用于 `renderer` 还是 `main`）→ 推荐方案与理由 → 必要的测试/回滚计划。

示例 prompt（依赖/分类优先）：
- "为 renderer 选择一个 HTTP client 库，列出备选项并按维护性与 Svelte 兼容性排序。"
- "评估将 `node-forge` 替换为 `webcrypto` 的影响，列出受影响模块、风险等级与回滚步骤。"

- 是否要求在每个 PR 中都显式标注维护者，还是仅对架构/模块变更要求？
