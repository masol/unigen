# 📐 工作流开发总览（开发者指南）

本页面向**构建**工作流的开发者，是所有开发者文档的入口和地图。

## 心智模型：你在开发什么

在 Unigen 中，工作流以**代码**形式存在（在界面中称为**蓝图**），可动态加载。选择代码而非配置 DSL，是一个刻意的决定：**为了让规划器（而不只是人）能够修改它**——规划器正是通过修改蓝图内容来改进工作流的。

"开发工作流"这件事本身也是一个工作流，因此它也是一种项目类型——当前版本中，**空白项目类型**即内建了这个能力。

## 两个核心工作面

打开项目后，你主要在两个地方工作：

### 反思助手（右侧边框展开，须有项目打开）

命令式交互入口，可执行命令、发布任务。

### 蓝图（工作流细节）

点击**蓝图**可查看和编辑工作流的全部细节。注意两点：

1. 你的手工编辑和规划器的自动改进作用在**同一份蓝图**上——这是"人工可介入 + AI 可持续改进"得以共存的载体。
2. 蓝图的用户配置界面由一棵 JSON 描述，见 [[动态面板 JSON 指南|Panel-JSON]]。

## 开发到发布的路径

```
空白项目 → 反思助手辅助开发 → 蓝图中打磨工作流 → 实际运行验证 → /export 导出为项目类型 → 发布给创作者
```

## 姿势二：规划器生成（演进中）

Unigen 的终极目标是让工作流的产生、改进、维护都由规划器接手。规划器路线分三阶段推进，参考其它文档。

三层共同的立场：**代码作骨、LLM 填充**——结构合法性（无环、依赖闭合、词表统一）由代码硬保证，LLM 只负责局部语义。这是对"长链条漂移零容忍"的落实。

## 架构速览

- **执行器**：调度与运行原子动作，捕获环境反馈，负责生产容错。
- **规划器**：把高层意图编译成结构合法的任务路径，通过修改蓝图落地。不干活，只编译。
- **执行历史**：OpenTelemetry 兼容采集，一切改进以真实运行记录为依据。
