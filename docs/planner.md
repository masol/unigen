# 规划器设计说明：图约束下的 HDDL 生成与路径解算

当前路径: docs/planner.md

状态: 思路验证阶段。核心数据管道（图构建 + 词表对齐）在开发中，解算层为过渡实现。

本文记录本系统在**任务规划**层面的设计取舍与演进方向。这里描述的是一条正在验证的路线，不是已经定型的结论；文中的对比基于逻辑推演和架构设计，而非工程实测数据。行业主流的 ReAct、Plan-and-Solve、LangGraph 状态机等方案已有大量社区实践和时间检验，本路线尚处早期，很多判断有待被真实任务推翻或修正。

---

## 一、系统定位：规划器与执行器分离

系统由两层构成，职责边界清晰：

- **执行器（Executor）**：完成具体的复杂任务，例如“从剧本到视频”这类多步骤、有中间产物的流水线。执行器负责原子动作的实际调度与运行。
- **规划器（Planner）**：不直接干活，而是**为执行器规划工作流**——把高层意图编译成一条结构合法、依赖清晰的任务路径，交给执行器去跑。

这样分离带来三个特性：

1. **人工可介入**。规划器产出的工作流是可读、可编辑的结构，人可以直接查看和修改，而不是黑箱里的一串 LLM 决策。
2. **AI 可持续改进**。同一份工作流后续可以交由 AI 基于历史表现继续优化。
3. **历史可追溯**。所有执行历史都会保存（通过配置 OpenTelemetry 兼容的端点采集），作为后续人工或 AI 改进工作流的依据。改进不靠猜，靠真实运行记录。

规划器的核心立场是：**它只负责静态路径的编译，不负责动态执行**。生产环境里的容错、重试、环境反馈，都收敛在执行器（外层工作流引擎）的循环中：

1. 执行器捕获当前环境的报错、异常或反馈；
2. 将最新状态浓缩为上下文，反哺给规划器；
3. 规划器基于新事实重新编译一条合法路径，交还执行器继续调度。

## 二、设计取舍：为什么走“代码作骨，LLM 填充”

这条路线的基本假设是：**结构由传统代码维护，语义由 LLM 局部填充**。图的拓扑、依赖关系、词表一致性交给 `graphology` 和代码逻辑保证；每一步“这个目标该拆成什么”交给 LLM。

下表是与主流 Planner 的对比。需要强调：右列是**设计意图**，不是已验证的结论，其中多数“优势”目前只是推断。

| 评估维度      | 主流 Agent Planner（ReAct / LangGraph） | 本路线（图约束 HDDL 生成）         |
| ------------- | --------------------------------------- | ---------------------------------- |
| 核心驱动      | LLM 语义直觉驱动                        | 代码骨架 + LLM 局部语义填充        |
| 规划/执行边界 | 规划与执行交织，走一步看一步            | 分离，规划器只编译路径             |
| 状态维护      | LLM 在长文本中模拟状态修改              | 不可变事实链，追加式引入时序       |
| 长链路        | 步数增多后易出现上下文漂移、幻觉        | 结构约束拦截逻辑环，**推断**上更稳 |
| 上下文成本    | 推理历史与反馈叠加，随步数增长          | 单步拆解上下文小，拓扑层内可并发   |
| 验证程度      | 已有大规模工程验证，容错高              | 早期阶段，对概念对齐精度要求高     |

关于成本一栏补充一句：ReAct 的 token 增长大致随步数近似二次增长。本路线上下文小是因为每次只做一个微型拆解，代价是调用次数变多、对“概念不漂移”的要求变高。

## 三、这条路线适合与不适合的场景

它不适合开放、模糊、缺乏规则约束的泛化任务——这类任务 ReAct 的语义直觉反而更合适。它的价值集中在**高合规、强逻辑、步骤多**的深度垂直场景：步骤越长、约束越硬，代码骨架和不可变状态带来的确定性收益越明显。

反过来说，对 3–5 步的简单任务，本路线要经历环境自举、构图、依赖校验等流程，明显偏重，性价比不如直接用 ReAct。选型时这一点要诚实对待。

## 四、图模型：AND-OR 图与受控递归

这是本路线区别于“普通依赖图”的关键，也是早期设计里最容易做错的地方。

### 4.1 用 AND-OR 图，而非纯 AND 依赖图

HTN/HDDL 里，一个复合任务（compound task）通常有**多种分解方法（method）**，规划时要在其中选一条。如果图里一个节点只允许一套分解，`(:method)` 就退化成了固定展开，失去了“方法选择”这一 HTN 的核心。

因此节点关系分两类：

- **OR 关系**：一个 compound 节点下挂多个候选 method——满足其中任一即可。
- **AND 关系**：一个 method 内部有若干 subtask，全部满足、且遵守顺序约束才算这个 method 成立。

### 4.2 递归是合法的，不能一刀切当死循环

早期一个直觉是“用 DAG 检测，发现环就拦截”。但 HTN 里递归 method 是**合法且常见**的，例如：

```
运输所有包裹 = 运一个包裹 + 运输剩下的所有包裹
```

如果用全局无环（`hasCycle`）一刀切，会把合法递归和真正的逻辑死循环一起杀掉。所以要区分两种“环”：

- **目标依赖层面**：可以要求无环，作为结构健全性检查。
- **method 分解层面**：允许受控递归，靠**深度上限 + 重复模式检测**来防死循环——即“同一分解模式在同一路径上重复超过 N 次”才判定为异常，而不是见环就拦。

工程上，加边前用 `graphology-dag` 的 `willCreateCycle(graph, src, dst)` 先判再加，避免“加边—检测—回滚”的浪费。是否放行，再叠加上面的递归深度/重复模式规则。

## 五、词表对齐：这条路线的工程天花板

HDDL 对谓词命名的精确度是代码级的。多阶段拆解中，LLM 很容易造出边界模糊的近义词（把 `code_ready` 写成 `is_code_finished`），一旦概念对齐失败，构图就会分裂成互不连通的碎片。这是本路线最现实的风险，靠代码层强控：

- **归一化**：所有状态/属性名进入图之前，强制转为小写蛇形命名（snake_case）。
- **词表复用**：调用 LLM 拆解时，把已有图节点 ID 作为 `allowed_vocabulary` 一并传入，引导 LLM 优先复用已有概念，而非另造新词。
- **近义词拦截**：LLM 可以返回原始文本，但代码层用一个对齐函数判定“新词是否等同于某个已有节点”。判定为等同则强制复用已有 ID，不新建节点；低置信度的情况留给人工确认或谨慎新建，而不是自动合并。

## 六、解算层：从 LLM 模拟到 Prolog 符号推导

规划器把高层意图拆成符合图结构的复合任务、方法与原子动作后，还需要“从 HDDL 找到一条可执行路径”。这一层的目标求解器分两个阶段推进，终态直接使用 Prolog。

### 6.1 为什么用 Prolog 作为 HTN 求解器

HTN 的分解搜索与 Prolog 的 SLD 归结（自顶向下、深度优先、带回溯）在结构上高度一致，映射几乎是一一对应的：

| HTN / HDDL 概念              | Prolog 对应                         |
| ---------------------------- | ----------------------------------- |
| 复合任务（compound task）    | 目标谓词（Goal）                    |
| 一个分解方法（method）       | 一条 Horn 子句（规则）              |
| 同一任务的多个 method        | 同名多子句——天然表达 OR（方法选择） |
| method 内的 subtask 序列     | 子句体的合取——天然表达 AND + 顺序   |
| 原子动作（primitive action） | 基例事实 / 带前置与效果的子句       |
| 任务间参数传递               | 合一（unification）自动完成         |

这带来几个直接好处：

- **AND-OR 搜索由引擎负责**。method 之间的 OR 回溯、subtask 之间的 AND 展开，都交给 Prolog，不必自己维护回溯栈。第四节的 AND-OR 图在这里正好落地为“同名多子句 + 子句体合取”。
- **转换层很薄**。graphology 里编译好的 HDDL 域，翻译成 Prolog 子句几乎是逐条对应，工程量集中在词表与谓词命名（第五节）而非搜索逻辑本身。
- **tabling 直接治递归**。现代 Prolog（如 SWI-Prolog）的 tabling（SLG 归结）会记忆化子目标：既避免重复推导，又能让朴素 Prolog 会死循环的左递归终止。这把第七节的“无限递归防御”从纯外层拦截，落到了求解器内部的真实机制上——外层 `graphology-dag` 负责编译期的结构健全性检查，tabling 负责求解期的终止保证，两者互补。
- **声明式、可审计**。同一份规则人可读、可改，契合“人工可介入、AI 可持续改进”的定位。
- **成熟实现**。复用 WAM/SWI 经年优化的回溯搜索，性能不用自己操心。

需要标注的边界：Prolog 默认是**满足性搜索**（找到一条合法路径即返回），不是**最优代价搜索**。因此它保证的是“存在且无冲突”，不保证“最优”；若需要最优或近优，要额外引入代价约束，或在 Prolog 之上叠加评估，或换用带启发式的符号 planner（如 SHOP2、PANDA、Fast Downward）。当前路线先追求“确定合法”，最优性留待后续。

### 6.2 阶段一（当前过渡）：LLM 模拟 Prolog 推演

- **做法**：暂不接入 Prolog 虚拟机。用带 `zod` 强约束的结构化输出，让 LLM 顺着不可变事实链，在思维链中模拟 Prolog 的回溯与推演，产出原子动作序列。这一步是 Prolog 求解器的**占位替身**，接口（输入 HDDL + 初始事实，输出动作序列）与阶段二保持一致，便于后续无缝替换。
- **能验证什么**：词表对齐是否稳定、结构是否合法（可拓扑排序）、数据管道是否连通。
- **不能保证什么**：LLM 模拟没有符号求解器的刚性约束，深层回溯下仍可能幻觉，解的正确性没有保证。过渡版的能力不等于终态能力。

### 6.3 阶段二（目标）：Prolog 符号推导（基于不可变状态）

- **求解机制**：HDDL 域翻译为 Prolog 子句后，以顶层任务为 Goal、初始事实为事实库，由 Prolog 的 SLD 归结完成 method 选择与 subtask 展开的搜索，输出一条 grounded 的原子动作序列。method 的 OR、subtask 的 AND 都由引擎回溯负责。
- **状态维护**：摒弃内存状态的破坏性修改，引入时间步，把世界状态看作只允许追加的事实链，例如 `at(car, shanghai, t0)` → 执行动作 → 追加 `at(car, beijing, t1)`。谓词带上时间步参数后，状态变化通过合一在时序上推进，副作用被消除，回溯时不需要撤销状态。
- **递归与终止**：开启 tabling 记忆化子目标，配合第四节的深度上限与重复模式检测，让合法递归（“运输所有包裹”式）正常展开，让病态闭环终止或被拦截。
- **兑现的价值**：复用符号引擎经年验证的回溯与搜索能力，沿 LLM 铺好的 HDDL 管道输出无冲突的动作序列。第二节表格里“确定性”一栏的收益，在这一阶段随 Prolog 接入而真正兑现。

## 七、需要重点防范的边缘情况

### 7.1 无限递归防御

- **风险**：LLM 生成 method 时可能引入逻辑闭环（A 依赖 B，B 又依赖 A），导致解算死锁或栈溢出。
- **防范**：如第四节所述，加边前 `willCreateCycle` 预判，叠加递归深度上限与重复模式检测。合法递归放行，异常闭环在编译期拦截并触发局部重试。区分“合法递归”与“死循环”是这里的关键，不能退回到见环就拦。

### 7.2 事实库膨胀导致的搜索爆炸

- **风险**：不可变事实持续追加，事实库随时间步线性增长，长周期任务下搜索空间可能膨胀。
- **防范**：设计“状态快照与剪枝”机制。执行器跑到安全里程碑时，把历史事实压缩为当前最简状态，丢弃不再影响未来的旧事实，重建一个新的 `t0` 基线。这一步同时也让历史记录（第一节的 OpenTelemetry 留存）有天然的分段边界。

## 八、当前状态与下一步

- **当前版本** 先实现一个G-Eval + MoA + Self-Refine结构的规划器，做为人工改造工作流的基础。先工程验证整个体系的价值--即便人工创建的长链条工作流，也是极有价值的。之后再引入本文描述的规划策略,尝试达成全自动规划及改进的目标。
- **已在做**：图构建、词表对齐（LexiconManager）、单步拆解的数据管道。
- **过渡实现**：解算层用 LLM 模拟，仅验证结构合法性与管道连通性。
- **待验证的关键假设**：AND-OR 图 + 受控递归能否稳定表达真实业务的 method 选择；词表对齐在多阶段拆解下的失败率有多高——这决定整条路线是否成立。
- HDDL 域翻译成 Prolog 子句后，tabling + 时间步事实链能否在真实业务的递归 method 上稳定终止并给出合法路径——这决定阶段二能否兑现确定性。
- **更远的方向**：接入真实符号解算器兑现确定性；基于 OpenTelemetry 采集的执行历史，让人工和 AI 持续改进已生成的工作流。

这份文档会随验证结果更新。凡是被真实任务证伪的判断，会明确标注修正，而不是悄悄改掉。
